### 2.2 引脚定义深度解析

```cpp
// 电机 A = 左轮
#define PIN_IN1_A 15    // PWM信号
#define PIN_IN2_A 13    // 方向信号

// 电机 B = 右轮
#define PIN_IN1_B 14    // PWM信号
#define PIN_IN2_B 25    // 方向信号
```

**为什么左右轮的IN1/IN2配置不同？** 左轮前进时：IN1_A=15给PWM, IN2_A=13给0（低电平）。右轮前进时：IN2_B=25给PWM, IN1_B=14给0。这种不对称是因为右轮电机的物理安装方向与左轮相反——两台电机面对面安装（输出轴朝向小车中心），从同一侧看，一台顺时针转=前进，另一台逆时针转=前进。代码使用`motor_right_set_pwm()`中的不同逻辑处理这一物理不对称。

```cpp
// 编码器方向（实测：前进时左轮cnt为正、右轮cnt为负）
#define LEFT_ENC_SIGN   1.0f
#define RIGHT_ENC_SIGN -1.0f
```

编码器A/B相接反导致右轮前进时计数方向与左轮相反。使用软件符号校准（而非重新焊接硬件接线）的好处是：可以在调参阶段快速修正，且不会因反复焊接损坏PCB焊盘。两个常量值来自阶段二的实测标定数据。

```cpp
// 传感器从左到右
#define TRACK_GPIO_1 34   // L2(最左)   ADC1_CH6
#define TRACK_GPIO_2 35   // L1          ADC1_CH7
#define TRACK_GPIO_3 32   // C(中心)     ADC1_CH4
#define TRACK_GPIO_4 33   // R1          ADC1_CH5
#define TRACK_GPIO_5 27   // R2(最右)    ADC2_CH7
```

**为什么GPIO27使用ADC2而非ADC1？** ESP32有两个SAR ADC外设。ADC1的8个通道（CH0~CH7）中只有部分映射到可用的GPIO。L2/L1/C/R1四个传感器恰好占用4个ADC1通道。R2若放在ADC1上会与其他通道的引脚冲突——GPIO27对应的ADC2_CH7是唯一可用的选择。但ADC2与WiFi功能共享——本系统主控板不使用WiFi，所以此冲突不存在。

### 2.3 控制参数深度解析

#### 2.3.1 外环PD参数

```cpp
#define KP_LINE            12000.0f
#define KD_LINE            4200.0f
#define NONLINEAR_TURN_GAIN 18900.0f
```

这三个数值不是任意选取的——它们之间存在深刻的物理关系：

**KP_LINE=12000的物理含义：** error∈[-2,+2]是归一化的无量纲偏差，而delta_speed的单位是pps（编码器脉冲/秒）。KP将"位置误差"转换为"速度差指令"。error=1.0（偏差一个传感器间距=9mm）产生12000pps的速差——在48000pps的巡航速度下，左右轮速分别为48000-12000=36000和48000+12000=60000pps，速度差异约25%。此差异在20mm宽的黑线上产生的转向角速度大约能在一秒内将车头转动约15°——这是经验上合适的转向灵敏度：太快则直线震荡，太慢则弯道不足。

**KD_LINE=4200与KP的比例关系：** KP:KD ≈ 12000:4200 ≈ 2.86:1。这个比例并非随意——在连续时间近似下，PD控制器的特征方程为 s + KP/KD = 0（假设纯惯性系统），时间常数 τ = KD/KP = 4200/12000 = 0.35秒。这意味着控制器对误差变化的"记忆"约350ms——比控制周期（20ms）长一个量级，比弯道通过时间（0.5~1秒）短。若KD过大（>8000），τ>0.67秒，转向响应迟钝，弯道偏差峰值增大；若KD过小（<2000），τ<0.17秒，阻尼不足，直道上轻微噪声就会引起持续震荡。

**NONLINEAR_TURN_GAIN=18900的设计分析：**

| error | P项贡献 | 非线性项贡献 | 非线性/P比 | 物理含义 |
|-------|---------|-------------|-----------|---------|
| 0.3 | 3600 | 1701 | 0.47 | 直线微调，非线性几乎不参与 |
| 0.5 | 6000 | 4725 | 0.79 | 轻微弯道，非线性温和补充 |
| 0.8 | 9600 | 12096 | 1.26 | 明显弯道，非线性超越P项 |
| 1.0 | 12000 | 18900 | 1.58 | 急弯入口，非线性主导 |
| 1.5 | 18000 | 42525→26000 | 1.44(被限幅) | 极弯，非线性被硬件保护截断 |
| 2.0 | 24000 | 75600→26000 | 1.08(被限幅) | 全偏，总输出受限于26000限幅 |

这个表格揭示了设计的精巧之处：在小偏差区（error<0.8），系统行为接近纯线性PD；在大偏差区（error>0.8），非线性项快速增长并最终成为转向力的主要来源。26000pps的限幅在error≈1.38时达到——此后的额外偏差不再产生额外转向力，这是对物理极限的尊重：如果黑线已经偏离到传感器阵列最外侧，说明偏差极大，但过大的转向指令可能导致车轮打滑或失稳，限幅起到最后的保护作用。

#### 2.3.2 弯道减速参数

```cpp
#define CURVE_SLOWDOWN_GAIN 22500.0f
#define CURVE_MIN_SPEED_PPS 19200.0f
```

这组参数的设计逻辑：

减速公式 `base = 48000 - 22500×|error|` 在 |error|=0 时 base=48000，在 |error|=1.28 时 base=48000-22500×1.28=19200=CURVE_MIN_SPEED_PPS。这意味着在偏差≥1.28时速度降至最低保护值。

为什么是19200（直道的40%）？三个考量：
1. **物理安全**：在40%速度下，即使完全冲出赛道，动能为(0.4)²=16%的全速动能，冲击力大幅降低。
2. **控制有效性**：19200pps对应的线速度约0.24m/s，此速度下20ms控制周期的小车位移约4.8mm，小于传感器间距9mm——意味着控制系统仍有足够的空间分辨率来纠正偏差。
3. **效率平衡**：若最低速度设得更低（如12000=25%），虽然更安全，但圈速会显著退化；若设得更高（如24000=50%），急弯中惯性力可能超过轮胎与地面的摩擦力极限，导致侧滑。

#### 2.3.3 内环PI参数的不对称设计

```cpp
#define KP_SPEED_L         0.040f
#define KI_SPEED_L         0.300f
#define KP_SPEED_R         0.038f
#define KI_SPEED_R         0.290f
```

为什么左右轮的参数不同？这是"设计补偿物理不对称"的典型案例：

- **物理根源**：左右电机来自同一批次，但因制造公差，右电机的绕组电阻略大（~5%），导致相同PWM下转速略低。
- **传动差异**：右轮传动齿轮的啮合间隙比左轮大约0.02mm（长期单方向运转的偏磨），摩擦力矩略大。
- **数学表达**：Km值直接量化了差异——左Km≈90pps/PWM, 右Km≈95pps/PWM。相同PWM下右轮获得的转速约低5.5%（557×90=50130 vs 557×95=52915pps...等等，Km值更大说明每单位PWM获得更多转速，所以右轮效率更高？）。实际上Km=速度/PWM，Km越大表示能效越高——右轮Km比左轮大约5.5%意味着相同PWM下右轮更快。这导致如果使用对称PI参数，右轮会"过度响应"，产生速度过冲。

具体的补偿逻辑：右轮Kp比左轮小5%（0.040→0.038），Ki比左轮小3.3%（0.300→0.290），总控制增益约小4~5%——恰好抵消Km约5.5%的效率优势。这是PID工程实践中"参数微调使闭环响应对称化"的标准做法。

#### 2.3.4 丢线策略参数

```cpp
#define LINE_LOST_HOLD_CYCLES 2         // 短期保持2周期=40ms
#define LOST_BASE_SPEED_PPS  13500.0f   // 丢线搜索速度
```

**为什么是40ms的保持窗口？** 在巡航速度48000pps下，40ms内小车的位移：

```
轮速 = 48000pps
轮周长 = π×65mm ≈ 204mm
轮转速 = 48000/11700 ≈ 4.10转/秒  (11700=390×30, 输出轴每圈脉冲数)
线速度 = 4.10×204 ≈ 837mm/s
40ms位移 = 837×0.04 ≈ 33.5mm
```

33.5mm的位移对比传感器总覆盖范围：5路传感器物理跨度36mm+黑线宽度20mm的传感范围=约56mm。40ms内黑线大概率仍在阵列范围内——这是"短期保持"策略成立的物理基础。

**为什么13500pps？** 这是丢线搜索速度=28%的全速。0.17m/s的线速度下，20ms位移约3.4mm——这一位移量远小于传感器总覆盖范围，确保在低速搜索中不会"跳过"黑线。

### 2.4 核心算法详解

#### 2.4.1 模拟归一化——从ADC到黑度

```cpp
static float normalize_sensor_to_black(int raw, int idx) {
    float white = (float)sensor_white[idx];
    float black = (float)sensor_black[idx];
    float denom = white - black;
    if (fabsf(denom) < 1.0f) return 0.0f;    // 分母保护
    float x = (white - (float)raw) / denom;
    return clampf_local(x, 0.0f, 1.0f);
}
```

**为什么是 `(white - raw)` 而不是 `(raw - black)`？**

这取决于传感器的物理极性。ITR9909在共射极接法下：白底→接收管导通深→集电极电压低→ADC读数低；黑线→接收管接近截止→集电极被上拉到VCC(3.3V)→ADC读数高。

所以 `raw_white ≈ 4000~4095`（高ADC = 白 = 高电压），`raw_black ≈ 800~1200`（低ADC = 黑 = 低电压）。因此 `white - raw` 给出的是"当前读数比白底暗了多少"——正值表示"不是纯白"，值越大越黑。

验证：raw=4000（白），white-raw=4095-4000=95, denom=2895, intensity=95/2895≈0.03（接近0=白）✓
验证：raw=800（黑），white-raw=4095-800=3295, denom=2895, intensity=3295/2895=1.14→clamp到1.0（纯黑）✓

**为什么用clampf_local而非直接截断？** 实测中的raw可能因ADC噪声而略微超出标定的[black, white]范围——例如某个传感器的实际black=850但标定值是1200（偏保守）。此时raw=850时intensity=(4095-850)/2895=1.12，超出[0,1]。clampf_local确保归一化输出永远在合法范围内，防止超出范围的偏差值在后继计算中放大。

#### 2.4.2 加权重心法——从强度分布到位置估计

```cpp
const float weight[5] = {-2.0f, -1.0f, 0.0f, 1.0f, 2.0f};

float weighted_sum = 0.0f;
float intensity_sum = 0.0f;
for (int i = 0; i < 5; i++) {
    d.intensity[i] = normalize_sensor_to_black(d.raw[i], i);
    weighted_sum += d.intensity[i] * weight[i];
    intensity_sum += d.intensity[i];
}
d.pos = weighted_sum / intensity_sum;
```

**这是一个"以强度为权重的离散位置加权平均"**。其数学本质是概率分布的一阶矩（期望值）。

举三个实际场景的例子：

**场景1：黑线居中**
```
sensor:   L2    L1     C     R1    R2
weight:   -2    -1     0     +1    +2
raw:     3950  3910   850   3890  3980
intensity:0.03  0.06  0.95   0.07  0.02
```
`pos = (0.03×(-2)+0.06×(-1)+0.95×0+0.07×1+0.02×2) / (0.03+0.06+0.95+0.07+0.02) = (-0.06-0.06+0+0.07+0.04) / 1.13 = -0.01/1.13 ≈ -0.01`

结果约为0，符合"黑线居中"的物理事实。

**场景2：黑线偏右一个间距**
```
sensor:   L2    L1     C     R1    R2
weight:   -2    -1     0     +1    +2
raw:     3990  3950   3920   860  3910
intensity:0.02  0.03  0.04   0.95  0.04
```
`pos = (0.02×(-2)+0.03×(-1)+0.04×0+0.95×1+0.04×2) / (0.02+0.03+0.04+0.95+0.04) = (-0.04-0.03+0+0.95+0.08) / 1.08 = 0.96/1.08 ≈ 0.89`

结果约0.89，接近+1.0（"偏右一个间距"），符合物理事实。

**场景3：边缘情况——两个传感器同时感应**
```
sensor:   L2    L1     C     R1    R2
weight:   -2    -1     0     +1    +2
raw:     3990  3880   870   880  3960
intensity:0.02  0.07  0.94   0.93  0.02
```
`pos = (0.02×(-2)+0.07×(-1)+0.94×0+0.93×1+0.02×2) / (0.02+0.07+0.94+0.93+0.02) = (-0.04-0.07+0+0.93+0.04) / 1.98 = 0.86/1.98 ≈ 0.43`

结果约0.43——这是一个"亚传感器间距"精度的位置估计：黑线在C(0)和R1(+1)之间约43%处。这正是模拟归一化+加权重心法的核心优势——5个离散传感器经过此算法处理后，能输出连续的位置估计，分辨率远超5个离散档位。

#### 2.4.3 EMA滤波与死区

```cpp
pos_filtered = POS_FILTER_ALPHA * pos_filtered +
               (1.0f - POS_FILTER_ALPHA) * line.pos;
```

注意代码中使用的是 `α×old + (1-α)×new` 的形式。通常EMA写作 `y[n] = α×x[n] + (1-α)×y[n-1]`，但这里α=0.20对应旧值的权重，所以新值的权重是0.80——这是一个"快速跟踪"的低通滤波器（时间常数τ≈20ms/0.80=25ms），相当于截止频率约1/(2π×0.025)≈6.4Hz。此频率选择能够：①充分跟踪弯道偏差变化（弯道通过时间0.5~1秒，对应1~2Hz频率分量）；②有效衰减ADC噪声的高频分量（>100Hz）。

死区处理 `if (fabsf(line_error) < LINE_ERROR_DEADBAND) line_error = 0.0f` 的0.08阈值对应约0.72mm的物理偏差。死区的工程价值不仅是消除电机微调嗡鸣声——更重要的是减少了PI积分器在微小误差上的无意义累积。如果没有死区，直道上持续存在error≈±0.02（噪声量级），经过100个周期（2秒）积分器会累积约0.02×100×0.3=0.6的值——虽不大但会产生持续的微小速度偏移。

#### 2.4.4 速度PI控制器的四个机制

**① 前馈补偿的物理建模**

```cpp
static float feedforward_left_pwm(float speed_ref) {
    float pwm = LEFT_FF_PWM_BASE +
                (speed_ref - LEFT_SPEED_AT_FF_PWM_BASE) / LEFT_KM_PPS_PER_PWM;
    return pwm;
}
```

前馈模型的建立过程（来自阶段二标定数据）：

| PWM | 左轮稳态速度(pps) | 右轮稳态速度(pps) |
|-----|-------------------|-------------------|
| 200 | 16500 | 17200 |
| 400 | 34500 | 35900 |
| 557 | 48000 | 50000 |
| 600 | 51900 | 54100 |
| 800 | 69900 | 73100 |
| 1000 | 87900 | 92100 |

对左轮做最小二乘线性拟合：speed = a + b×PWM，得到 b≈90pps/PWM, a≈-2133（截距，对应克服静摩擦所需的最小PWM）。

选取基准工作点(PWM=557, speed=48000)作为线性化的展开点：
```
FF_PWM = 557 + (speed_ref - 48000) / 90
```
验证：speed_ref=48000→FF=557 ✓；speed_ref=66000（直道加速全速）→FF=557+18000/90=757 ✓

**② 死区补偿——跳过静摩擦区**

```cpp
static float apply_signed_deadzone(float pwm, float ref_speed,
    float forward_dead_pwm, float pwm_min, float pwm_max) {
    if (ref_speed > 0.0f) {
        if (pwm > 0.0f && pwm < forward_dead_pwm) {
            pwm = forward_dead_pwm;  // 跳跃过死区
        }
    }
    // ...
}
```

电机在极低PWM下存在"有电流但无转速"的死区——电流产生的电磁力矩不足以克服静摩擦力。实测左轮需PWM≥270、右轮需≥295才开始转动。死区补偿将PWM从[0, 死区]区域的任意值映射到"死区边界"——这本质是分段线性化的前向增益变换：

```
无补偿时：  0~270→0rpm, 270~1023→线性
补偿后：    0→0rpm, 1~270→死区边界转速, 270~1023→线性
```

**③ 条件积分抗饱和**

```cpp
float unsat_pwm = ff_pwm + pi->kp * error + pi->ki * candidate_integral;
bool high_sat = (unsat_pwm > pwm_max) && (error > 0.0f);
bool low_sat  = (unsat_pwm < pwm_min) && (error < 0.0f);
if (!high_sat && !low_sat) { pi->integral = candidate_integral; }
```

条件积分的逻辑直觉："如果控制器已经100%输出但效果还不够（误差与饱和方向一致），再增加积分只会让问题更糟"。

具体四种场景：
- 饱和上界(high_sat) + error>0（需要更大输出但已达上限）→ **停止积分**（防止windup）
- 饱和上界 + error<0（输出已过大，需要减小）→ **继续积分**（让积分减小以退出饱和）
- 饱和下界(low_sat) + error<0（需要更小输出但已达下限）→ **停止积分**
- 饱和下界 + error>0（输出已过小，需要增大）→ **继续积分**

**④ 左右独立参数**

```cpp
static PI_Controller pi_left  = { .kp = 0.040f, .ki = 0.300f, ... };
static PI_Controller pi_right = { .kp = 0.038f, .ki = 0.290f, ... };
```

使用结构体封装PI控制器状态的设计模式，使得两个车轮的控制逻辑完全共享同一段代码(`pi_speed_update`)，仅通过不同的参数实例化来实现差异补偿。`integral_limit = ±6000` 意味着积分器最多能贡献 Ki×6000 = 0.300×6000 = 1800 PWM单位（左轮）。在48000pps巡航时PWM约557，积分的最大贡献约323%——这是一个相当宽的范围，允许控制系统在启动阶段（速度从0加速到48000，需要大幅PWM输出）有足够的积分推动力。

#### 2.4.5 主循环时序保证

```cpp
TickType_t last_wake = xTaskGetTickCount();
while (1) {
    vTaskDelayUntil(&last_wake, pdMS_TO_TICKS(CONTROL_DT_MS));
    // ... 20ms控制逻辑 ...
}
```

`vTaskDelayUntil` 是FreeRTOS中保证精确周期的API——它记录上一次唤醒时间，计算下一次唤醒时间，直接延迟到该时刻，不被任务执行时间所影响。即使某次循环中的控制逻辑耗时波动（如某次UART缓冲区数据较多导致解析耗时略长），下一次的唤醒时间仍然是恰好20ms后——这保证了50Hz控制频率的长期精确性。

实测抖动分析：在正常负载下（主控CPU~25%），周期抖动<300μs（1.5%），对控制精度影响可忽略。最坏情况下（DEBUG_PRINT_EVERY_N=1，每周期打印约300字节到115200串口），周期可能扩展到约22ms（+10%）。生产版本将DEBUG_PRINT_EVERY_N设为5，将打印开销分摊到每100ms一次，有效避免了此问题。

### 3.2 I²C自动发现流程详解

```cpp
static bool findMpu() {
    const int pin_sets[][2] = {
        {21, 47},    // 第一组：标准接线 SDA=GPIO21, SCL=GPIO47
        {47, 21},    // 第二组：交换接线 SDA=GPIO47, SCL=GPIO21
    };

    for (int p = 0; p < 2; p++) {
        int sda = pin_sets[p][0];
        int scl = pin_sets[p][1];
        if (!i2cStartDriver(sda, scl)) continue;

        // 全总线扫描（地址1~126），打印所有检测到的I²C设备
        for (int addr = 1; addr < 127; addr++) {
            if (probeAddr((uint8_t)addr)) {
                printf(" 0x%02X", addr);
            }
        }

        // 尝试MPU6050的已知地址
        for (uint8_t addr : MPU_ADDRS) {  // MPU_ADDRS = {0x68, 0x69}
            uint8_t who = 0;
            esp_err_t err = readRegs(addr, 0x75, &who, 1);  // 读WHO_AM_I寄存器
            if (err == ESP_OK && (who == 0x68 || who == 0x69)) {
                active_addr = addr;
                return true;
            }
        }
    }
    return false;
}
```

**为什么使用ESP-IDF原生I²C驱动而非软件模拟？** 此固件是纯姿态检测版，不启动摄像头。S3板的硬件I²C控制器此时空闲（没有被SCCB占用），使用硬件I²C可以获得：①400kHz的标准Fast Mode速率（远快于软件I²C的~25kHz）；②硬件ACK/NACK检测和自动重试；③DMA支持（可选，减少CPU参与）。

**WHO_AM_I验证的必要性：** I²C总线扫描仅能确认"某地址有设备应答"，但不能确认"该设备是MPU6050"。读寄存器0x75（WHO_AM_I）获取芯片ID号——MPU6050的出厂默认值为0x68（部分变体为0x69）。如果没有此验证，可能将其他I²C设备（如某些OLED屏、温度传感器）误识别为MPU6050。

### 3.3 带迟滞的状态机——完整状态转移图

```
                          |gz| < 5°/s
          ┌─────────────────────────────────────────────┐
          │                                             │
          ▼                                             │
    ┌──────────┐    straight_count >= 12     ┌──────────┐
    │  不确定   │ ──────────────────────────→ │  直道状态 │

          │  curve_count++               curve_count++
          │                                        │
```

**每个常量的选择和含义：**

| 常量 | 值 | 物理含义 |
|------|-----|---------|
| `STRAIGHT_GZ_DPS_TH` | 5.0°/s | Z轴角速度低于此值认为在直行（安静的手持晃动约2~3°/s） |
| `STRAIGHT_GZ_DPS_EXIT_TH` | 8.0°/s | Z轴角速度高于此值认为在转弯（明显可感知的转弯速率） |
| `STRAIGHT_CONFIRM_CYCLES` | 12 (240ms) | 直道需连续12个周期确认为真直道（排除瞬时角速度下降的假象） |
| `STRAIGHT_LOST_CYCLES` | 5 (100ms) | 弯道仅需5个周期确认为真弯道（安全优先——宁可多判弯道、不能漏判） |

**为什么直道确认(240ms)比弯道确认(100ms)更慢？** 这是安全策略的体现——直道的代价是"触发加速"，弯道的代价是"取消加速"。如果直道确认过快（如仅5周期=100ms），弯道中短暂的角速度下降（如S弯的拐点处角速度瞬变为0）可能被误判为直道→触发加速→在即将到来的下一个弯中速度过高而冲出。而弯道确认的延迟仅100ms——如果真的是弯道，晚100ms取消加速最多让小车在弯道入口处多保持了100ms的高速（约前进8.4cm），影响可控。

## 四、camera.cpp —— S3视觉+姿态融合固件

### 4.1 核心架构决策

这是S3板的完整功能固件，在一颗芯片上同时运行两个并行的感知任务：

```
主循环 (20ms周期):
  ┌──────────────────────────────────────────────┐
  │                                              │
  │  read_motion(&motion)  ──→  send_gyro(motion.gz)   │

  │                                              │

  │  (DVP DMA, ~16ms)          (CPU, ~3-5ms)           (UART)         │
  │                                              │
  │  LED更新                                     │
  │                                              │
  └──────────────────────────────────────────────┘
```

摄像头帧获取(`esp_camera_fb_get`)和MPU6050读取(`read_motion`)在时间上是串行的（均在同一个while循环中），但摄像头使用DVP+DMA传输（CPU仅触发，数据由硬件搬运），MPU6050使用软件I²C（CPU参与但数据量极小——仅2字节），总CPU开销在50Hz下约45~55%。

### 4.2 软件I²C的完整实现 —— 一个微型I²C主机

```cpp
// 时序参数
static void sw_i2c_delay(void) {
    esp_rom_delay_us(20);  // 约25kHz SCL频率
}
```

软件I²C的完整操作序列（以读取MPU6050寄存器0x3B的14字节加速度+陀螺仪数据为例）：

```
1. START: SDA=1, SCL=1 → SDA=0 → SCL=0
2. 写设备地址+W: 发送8bit (0x68<<1 | 0 = 0xD0), 检查ACK
3. 写寄存器地址: 发送8bit (0x3B), 检查ACK
4. REPEATED START: SDA=1, SCL=1 → SDA=0 → SCL=0
5. 写设备地址+R: 发送8bit (0x68<<1 | 1 = 0xD1), 检查ACK
6. 读14字节: 每字节8个时钟, 前13字节发ACK(0), 最后字节发NACK(1)
7. STOP: SDA=0 → SCL=1 → SDA=1
```

总共需要: 1(START)+9(地址+W)+9(寄存器)+1(RE-START)+9(地址+R)+14×9(读14字节)+1(STOP) = 157个bit时间。在25kHz下每个bit约40μs，总计约6.3ms。与20ms轮询周期相比，I²C通信占总时间的约31%。

**为什么软件I²C不使用DMA或中断？** 因为bit-bang I²C的每个bit都是CPU显式控制的（设置SDA电平→翻转SCL→读SDA），本身不涉及硬件外设，因此没有对应的DMA通道。如果需要更高的I²C速率（如400kHz），必须改用硬件I²C——但此固件中硬件I²C被摄像头SCCB占用。这就是"软件I²C独立于SCCB"这一架构决策的代价：CPU开销略高，但避免了资源冲突，且对20ms周期来说完全可接受。

#### 4.3.2 亮度提取（RGB565→灰度）

```cpp
static inline uint8_t pixel_luma(const camera_fb_t *fb, int x, int y) {
    const uint8_t *p = fb->buf + ((y * fb->width + x) * 2);
    uint16_t rgb = ((uint16_t)p[0] << 8) | p[1];
    uint8_t r = (uint8_t)(((rgb >> 11) & 0x1F) << 3);  // 5bit→8bit
    uint8_t g = (uint8_t)(((rgb >> 5)  & 0x3F) << 2);  // 6bit→8bit
    uint8_t b = (uint8_t)((rgb & 0x1F) << 3);           // 5bit→8bit
    return (uint8_t)(((uint16_t)r * 30 + (uint16_t)g * 59 + (uint16_t)b * 11) / 100);
}
```

RGB565格式：红色5bit+绿色6bit+蓝色5bit=16bit。先扩展到8bit精度，然后使用ITU-R BT.601亮度公式：Y = 0.299×R + 0.587×G + 0.114×B。代码中的整数近似(30/100≈0.300, 59/100≈0.590, 11/100≈0.110)与此标准公式吻合。

#### 4.3.3 自适应阈值计算

```cpp
int vmin = 255, vmax = 0;
for (int y = row_start; y < row_end; y += row_step) {
    for (int x = col_start; x < col_end; x += 2) {
        int v = pixel_luma(fb, x, y);
        if (v < vmin) vmin = v;
        if (v > vmax) vmax = v;
    }
}
int contrast = vmax - vmin;
if (contrast < 12) return out;  // 对比度不足→放弃本帧
int threshold = (vmax + vmin) / 2;  // 中值作为黑白分界
```

**为什么使用(vmax+vmin)/2而非OTSU等更复杂的方法？** OTSU（大津法）需要计算灰度直方图并搜索最大化类间方差的阈值，在ESP32-S3上计算量较大（需遍历256级灰度×判断每个像素）。而(vmax+vmin)/2仅需一次全局扫描即可获得。在白底黑线的简化场景下，灰度直方图呈现清晰的双峰分布（一个白峰在200~250范围，一个黑峰在20~80范围），中值法能有效分离双峰——性能与复杂度之比最优。对比度<12的阈值是对极低光照条件的保护——此时黑白分离不可靠，放弃本帧比用错误数据控制更安全。

#### 4.3.4 逐行最长黑段检测

每行的处理逻辑：

```
对于行y:
  扫描 x = col_start 到 col_end
  维护当前黑色段 [cur_start, cur_end] 和 全局最佳黑色段 [best_start, best_end]
  
  当遇到黑像素(pixel_luma < threshold):
    if 新段开始: cur_start = x
    cur_len++
  
  当遇到白像素或行末:
    if cur_len > best_len: 更新best
    cur_len = 0, cur_start = -1
  
  后处理:
    if best_len < 3: 丢弃 (噪声)
    if best_len > 70%行宽: 丢弃 (阴影/大黑块)
    
  该行输出: center_x = (best_start + best_end) / 2
```

**为什么选择"最长"黑段而非"第一个"黑段？** 赛道上可能存在多个暗区域——黑线本身、地面上的污渍、赛道外的阴影。选择最长的一段利用了"黑线通常是画面中最连续、最粗的暗色结构"这一先验知识。

**为什么过滤宽度>70%行宽的黑段？** 当小车偏出赛道、整片画面都是暗色地面时，全行都可能被判为"黑"，此时宽度接近100%行宽。此过滤器防止了这种误判。

#### 4.3.5 三区域加权融合的数学推导

```cpp
int group = 0;  // 远区 (顶部1/3)
if (rel > rows_span * 2 / 3) group = 2;      // 近区 (底部1/3)
else if (rel > rows_span / 3) group = 1;     // 中区 (中间1/3)

// 综合权重 = 纵向权重(越近越大) × 横向权重(黑段越长越可信)
float y_weight = 1.0f + (float)(y - row_start) / (float)rows_span;  // ∈[1.0, 2.0]
float len_weight = clampf_local((float)best_len / 16.0f, 0.5f, 2.0f);  // ∈[0.5, 2.0]
float weight = y_weight * len_weight;  // ∈[0.5, 4.0]
```

三区域融合的物理直觉可以用一个场景说明：

```
    远区(顶部1/3)            中区(中间1/3)          近区(底部1/3)
  黑线偏左 → 弯道预判    黑线逐渐回到中间 → 过渡    黑线在中间 → 当前无需转向
  far_p = -0.3             mid_p = -0.1             near_p = 0.0
```

非融合（仅用全图平均）：`all_p = (-0.3-0.1+0.0)/3 = -0.13`。系统会轻微左转——这是错误的，因为近处的黑线在正中，说明当前不需要转向。

融合（近区75%主导）：`P = near_p×0.75 + mid_p×0.25 = 0.0×0.75 + (-0.1)×0.25 = -0.025`。输出接近零——"虽然远处看到弯道在左边，但近处黑线还在正中，我现在不需要转向"。

这就是三区域加权的核心价值：**区分"需要现在做的"和"需要将来做的"**。

方向偏差A的计算：`A = far_p - near_p`。物理含义——如果远区偏差与近区偏差同号，说明黑线一直偏在同一侧（可能是直线但小车本身偏了，需要持续转向修正）；如果异号且差值大，说明有弯道趋势（远处偏左、近处居中→即将进入左弯）。

#### 4.3.6 质量评估系统

```cpp
float row_q = clampf_local((float)valid_rows / (expected_rows * 0.65f), 0.0f, 1.0f);
float contrast_q = clampf_local((float)(contrast - 12) / 70.0f, 0.0f, 1.0f);
float width_q = (out.w >= 0.010f && out.w <= 0.50f) ? 1.0f : 0.35f;
out.q = clampf_local(row_q * 0.55f + contrast_q * 0.35f + width_q * 0.10f, 0.0f, 1.0f);
```

三项评分的权值（55%+35%+10%）反映了各因素的置信度贡献：
- **行覆盖率(55%)**——最可靠的信号。如果55行中只有5行检测到黑线，无论这些行的黑线多清晰，整体结果不可信（可能只是几条随机暗线）。
- **对比度(35%)**——重要但不如行覆盖可靠。高对比度通常意味着好光照，但也可能是强烈的方向性光造成的影子。
- **黑线宽度(10%)**——辅助信号。宽度在期望范围内加分，超出则减分（但不完全否定，因为宽度的期望值本身受摄像头角度影响）。

丢线阈值Q<0.22的选择：此值约等于"50%行覆盖×0.55 + 最小对比度(12)×0.35/70≈0.06×0.35≈0.02 + ... + 宽度适中/不匹配"，即要求在行覆盖和对比度都不能太差的情况下才认为在线。

#### 4.3.7 全屏直道检测 —— 10+条件的AND融合

```cpp
bool whole_screen_straight =
    !out.lost &&                    // ① 不丢线
    out.q >= 0.42f &&               // ② 整体质量良好
    row_q >= 0.58f &&               // ③ 行覆盖率充足
    center_row_ratio >= 0.70f &&    // ④ 70%以上的有效行在中心(偏差≤0.26)
    out.valid_rows >= 9 &&          // ⑤ 至少9行有效（约占总行数的16%）
    out.w >= 0.020f &&              // ⑥ 黑像素比例下界（不能太细）
    out.w <= 0.52f &&               // ⑦ 黑像素比例上界（不能太粗）
    fabsf(out.p) <= 0.65f &&        // ⑧ 总体位置偏差不能太大
    fabsf(out.a) <= 0.35f &&        // ⑨ 方向偏差小（直线特征）
    fabsf(out.c) <= 0.45f;          // ⑩ 曲率信号小（非弯道）
```

这12个条件每个都排除了一种"非直道但可能被误判"的情况：
- 条件②③防止低质量帧被误判
- 条件④确保黑线从头到尾在中间（真正的直道特征）
- 条件⑤防止只有少数行有效（如外界短暂遮挡）
- 条件⑥⑦确保黑线宽度在物理合理范围
- 条件⑧⑨⑩确保偏差、方向和曲率都处于"直道"的特征范围

连续10帧确认后才触发BOOST——在50Hz下相当于200ms的确认延迟。这确保了一个短暂的"看起来像直道"的片段不会触发加速。

### 4.4 上电初始化时序的工程考量

```cpp
void app_main(void) {
    led_init();           // ① LED最先初始化——用户可以看到上电指示
    link_init();          // ② UART初始化——立即建立与主控的通信通道
    send_boot_status(-0.90f, 0.10f);   // ③ 通知主控"S3板已启动"
    sleep_ms(200);
    send_boot_status(-0.70f, 0.12f);   // ④ 渐进状态更新
    mpu_ready = mpu_init();            // ⑤ MPU6050先于摄像头初始化
    send_boot_status(-0.45f, mpu_ready ? 0.30f : 0.08f);
    bool cam_ok = (camera_init() == ESP_OK);  // ⑥ 摄像头后初始化
    send_boot_status(cam_ok ? 0.00f : -0.50f, cam_ok ? 0.25f : 0.05f);
    // ...进入主循环...
}
```

**为什么MPU6050先于摄像头初始化？** 软件I²C初始化仅涉及GPIO配置（开漏输出+上拉），不依赖其他外设。摄像头初始化需要通过SCCB向OV2640发送一系列配置命令（分辨率、帧率、白平衡、曝光等），耗时较长（~200-500ms）。如果摄像头先初始化，主控板在摄像头的初始化期间收不到任何S3数据（包括陀螺仪数据），导致250ms后gyro_fresh超时。MPU6050先初始化确保摄像头准备期间陀螺仪数据已经可用。

**boot_status的含义：** send_boot_status(float p, float q)函数发送一个模拟的CAM帧——函数名中的p和q不是视觉偏差P和质量Q，而是"进度指示"：
- p从-0.90→-0.70→-0.45→0.00（负值表示"尚未就绪"，0表示"完成"）
- q从0.10→0.12→0.30→0.25（递增表示"进度增长"）

主控端可根据接收到的CAM帧中P的极性（负=未就绪）决定是否在S3就绪前降级运行（仅使用光电传感器，忽略视觉数据）。

## 六、系统关键设计决策分析

### 6.1 为什么主控不直接读MPU6050？

主控板（ESP32 V1）完全有能力通过I²C直接读取MPU6050——它有闲置的I²C控制器。但设计选择让S3板读取MPU6050并通过UART转发数据，理由有三：

1. **引脚资源**：主控板已使用大量GPIO——5路ADC、2路编码器(4pin)、4路PWM、2路UART。如果再加I²C(2pin)，GPIO分配会更紧张。S3板本就有MPU6050（板载），零额外引脚开销。

2. **CPU负载分担**：主控板的50Hz PID控制已是硬实时任务。如果每20ms再增加I²C读取（约2~6ms额外延迟），在最坏情况下控制周期可能延长至22~26ms。将MPU6050读取卸载到S3保持了主控的时序纯净。

3. **架构对称性**：S3板天然是"感知协处理器"——处理摄像头和IMU两种传感器数据，统一通过UART输出。这形成清晰的"感知-控制"物理边界。

### 6.2 为什么丢线后不原地旋转搜索？

丢线策略中的"沿原方向低速搜索"而非"原地旋转360°扫描"的设计理由：

1. **赛道连续性**：黑线是连续曲线，小车因惯性不可能瞬间反向。上一刻黑线在右侧（error>0），说明小车刚离开一个右弯或正在直线偏右——最合理的搜索方向就是右侧。

2. **时间成本**：原地旋转耗时2-3秒（一个完整的360°扫描），而"沿原方向低速前进"通常在0.5-1秒内重捕黑线。

3. **位置保持**：旋转会导致小车在赛道上留下一个"旋痕"（轮胎在地面打滑），损坏赛道表面，且旋转停止后位置大幅偏移（可能偏出0.5m以上）。

### 6.3 前馈模型为什么选线性而非查表？

速度→PWM的关系在理论上不是完美线性的（电机有非线性区、齿轮效率随转速变化）。但实测数据R²>0.98表明在本系统的工作范围内（19200~66000pps，对应PWM约270-757），线性近似足够精确。查表法虽能捕捉非线性，但带来三个劣势：表的大小×精度=内存开销、边界外需插值、修改时需重新标定全表。线性模型仅需两个参数(Km和基准点)，调参直观。
